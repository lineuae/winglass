# winglass — developer guide

Guide for resuming work on this project.
Keep it current as the architecture changes.

## What winglass is

A Windows process manager focused on visibility. What Task Manager doesn't
show, and what Process Explorer stopped showing well twenty years ago.

The core value is **signature-based trust**. Every executable and DLL is
Authenticode-verified. Files signed by "Microsoft Windows" are marked green;
anything else — including a hostile file renamed to `svchost.exe` and dropped
in `C:\Windows\System32\` — is not. The tool never trusts a process by its
name or path.

Everything else (loaded DLLs, network sockets, disk I/O, environment vars,
SHA-256 hashes, sparklines) is built on top of that principle.

Rust backend + SvelteKit 5 frontend + Tauri v2 shell. Single ~10 MB binary,
no runtime, no cloud.

## Toolchain requirements (critical)

**Rust must resolve to the MSVC toolchain.** Not GNU/mingw. The dependency
graph (Tauri + windows-rs + sysinfo + sha2 + a large surface of the windows
crate) exceeds the mingw linker's export ordinal limit (~65 K); a mingw build
dies at link with:

```
ld.exe: error: export ordinal too large: 113249
```

If you see that: your `cargo` is resolving to a standalone Rust GNU install,
not to the rustup shim in `%USERPROFILE%\.cargo\bin`. Two fixes:

- **Immediate:** `$env:Path = "$env:USERPROFILE\.cargo\bin;" + $env:Path` for
  the current shell.
- **Permanent:** uninstall the "Rust stable GNU" entry from Programs and
  Features so only rustup's MSVC toolchain is on PATH.

Also required:
- Node.js 20+
- Visual Studio 2022 Build Tools with the **VC++ workload** (for `link.exe`)
- WebView2 — preinstalled on Windows 11, comes with recent Edge on Windows 10

## Architecture

### Command flow

The frontend polls `list_processes` every 1 s. When a row is clicked, it
polls `get_process_detail(pid)` every 1 s while the detail panel is open.
`kill_process(pid)` is fire-and-forget.

Rust owns all state. The frontend is a thin view: it doesn't cache derived
data, it just renders whatever the last command returned.

```
Svelte page  ─── invoke ──▶  Tauri command  ─── Mutex<AppState>
       ▲                          │                    │
       │                          ▼                    ▼
       └── JSON response ────  Rust modules ──▶  Windows APIs
                                                  + workers
```

### AppState (`lib.rs`)

Wrapped in `Mutex<>` and managed via `tauri::State`. Owns:

- `sysinfo::System` + `Users` — the process snapshot source
- Three HashMaps per-pid: `io_state` (last sample + timestamp for delta),
  `cpu_history`, `mem_history` (VecDeque, 60 samples each)
- `exe_path_cache: HashMap<u32, Option<String>>` — pid → resolved path
- `sig_cache: HashMap<String, Option<verify::SigStatus>>` — **lowercase path**
  → verification result. `None` means enqueued but not yet resolved. See
  "Signature cache design" below.
- `dns_cache: HashMap<IpAddr, Option<String>>` — same shape as sig_cache
- `sha_cache: HashMap<String, Option<String>>` — SHA-256 hex, cached per path
- `dll_cache: HashMap<u32, Result<Vec<String>, String>>` — pid → DLL paths
  or an error label (`access denied` or `protected process (PPL, Antimalware)`)
- Sender / Receiver pairs for the sig and DNS workers

Two background threads:
1. **Signature verifier** — `verify::start_worker()` — reads paths from an
   mpsc channel, runs `WinVerifyTrust` on each, sends back a `SigStatus`.
   Verify is 30-100 ms per file and can occasionally hang on network cert
   revocation, so it must not block the mutex.
2. **Reverse DNS resolver** — `dns::start_worker()` — same pattern for
   `GetNameInfoW` calls.

### The three-tier exe path resolution

`sysinfo::Process::exe()` returns `None` for a lot of privileged services on
non-admin runs — this is what makes 23 out of 27 svchosts look pathless. To
fix that, we fall back:

```rust
sysinfo_exe
    .or_else(|| win::process_image_path(pid))   // QueryFullProcessImageNameW
    .or_else(|| win::nt_image_path(pid))         // NtQuerySystemInformation(88)
```

Tier 3 (`SystemProcessIdInformation`) does not require `OpenProcess` at all —
the kernel just tells us. Returns an NT device path
(`\Device\HarddiskVolume3\...`) which we convert to Win32 via a drive-letter
map built from `QueryDosDeviceW` + `GetLogicalDriveStringsW`. This is how
Process Explorer / sigcheck resolve PPL exe paths.

### Signature verification pipeline (verify.rs)

Two-step, catalog-aware:

1. Run `WinVerifyTrust` on the file directly with `WINTRUST_ACTION_GENERIC_VERIFY_V2`.
2. If it returns `TRUST_E_NOSIGNATURE`, calculate the file's Authenticode hash
   with `CryptCATAdminCalcHashFromFileHandle2`, look it up via
   `CryptCATAdminEnumCatalogFromHash`, get the catalog file path via
   `CryptCATCatalogInfoFromContext`, then run `WinVerifyTrust` on the catalog
   itself. If the catalog is signed, the file inherits that trust.

Without step 2, ~50 % of Windows system DLLs look unsigned. Sigcheck does the
same dance.

While the WinTrust state handle is still open (between VERIFY and CLOSE
state actions), we call `WTHelperProvDataFromStateData` →
`WTHelperGetProvSignerFromChain` → `WTHelperGetProvCertFromChain` →
`CertGetNameStringW(CERT_NAME_SIMPLE_DISPLAY_TYPE)` to extract the signer's
Common Name. Then we close the state.

**Trust badge rule:** `signer.starts_with("Microsoft Windows")` → OS-signed
(green). Anything else that's Valid → generic signed (neutral). This is
deliberately narrower than `"Microsoft "` — Office and Azure use different MS
signing certs, and we don't want them wearing the OS badge.

### Signature cache is case-insensitive

Windows paths are case-insensitive at the OS level; a HashMap is not.
`sysinfo` and `QueryFullProcessImageNameW` don't always agree on casing —
one might return `C:\Windows\...`, the other `C:\WINDOWS\...`. We lowercase
paths at every insert / lookup site in `AppState::sig_cache`. This one bug
caused 23/27 svchosts to look non-green for several minutes during the
original TUI build.

### PPL detection

Some processes (Windows Defender's MsMpEng, csrss, LSASS with RunAsPPL, etc.)
run at Protected Process Light or Protected Process level. Even
Administrator cannot enumerate their DLLs — this is a by-design Windows
security boundary that only a signed kernel driver can cross.

Rather than showing `access denied (0x5)`, we call
`NtQueryInformationProcess(ProcessProtectionInformation, 61)`. The one-byte
`PS_PROTECTION` return encodes:

- `protection_type` (bits 0-2): 0 = none, 1 = PP, 2 = PPL
- `signer_type` (bits 3-6): 1 Authenticode, 2 CodeGen, 3 Antimalware,
  4 Lsa, 5 Windows, 6 WinTcb, 7 WinSystem, 8 App

The DLL error label becomes `protected process (PPL, Antimalware)` in yellow.

### Threads (threads.rs)

`NtQuerySystemInformation(SystemProcessInformation, 5)` returns a linked
list of `SYSTEM_PROCESS_INFORMATION` records; the thread table for each
process follows its record inline in the same buffer. We walk the list,
match the record with the target `UniqueProcessId`, and read the
following array of `SYSTEM_THREAD_INFORMATION`.

Rather than mirror the ~40-field NT process struct (which drifts subtly
between builds — new "reserved" slots become named fields over time), we
read only the fields we need at their fixed x64 offsets and treat the
rest as opaque padding:

- `NumberOfThreads` @ 4
- `UniqueProcessId` @ 80
- thread array @ 256 (stable on Vista+; the trailing counters shift but
  the header size does not)

Per-thread we surface: TID, `ThreadState`, `WaitReason` (only meaningful
when `ThreadState == Waiting`), user + kernel time, priority, and
context switch count. The state / wait-reason enums are hardcoded — they
haven't changed since Vista.

The whole-system walk is cheap (~5 ms on a typical desktop) so we do
this fresh on every `get_process_detail` call rather than caching. Thread
state is transient by nature and stale data would be worse than a small
extra call.

### Handles (handles.rs)

`NtQuerySystemInformation(SystemExtendedHandleInformation, 64)` returns
the entire kernel handle table. Class 16 (`SystemHandleInformation`)
truncates the owning PID to 16 bits and mangles handles > 64K; class 64
uses `ULONG_PTR` throughout and is the right one for modern systems.

Each entry is 40 bytes: `Object`, `UniqueProcessId`, `HandleValue`,
`GrantedAccess`, `CreatorBackTraceIndex`, `ObjectTypeIndex`,
`HandleAttributes`, `Reserved`. We keep the ones matching the target
PID.

**Type-name resolution.** The entry gives us `ObjectTypeIndex`, an
opaque per-boot integer. To turn it into a string like "File" or
"Section" we need `NtQueryObject(handle, ObjectTypeInformation)` — but
that requires the handle to live in *our* address space, so first we
`OpenProcess(PROCESS_DUP_HANDLE) + DuplicateHandle`. Type indices are
stable per-boot within one machine, so we resolve one exemplar per
index and cache. A process with 500 File handles = one `NtQueryObject`
call, not 500.

**Why no object names.** `NtQueryObject(ObjectNameInformation)` on a
handle to a named pipe or certain device objects can wander into driver
code that blocks indefinitely. Process Explorer solves this by running
the query in a sacrificial worker thread that gets terminated if it
doesn't return in ~100 ms. That plumbing is a fair chunk of code; for
the first cut we surface type + access mask + handle value only.

**Cached per pid** like `dll_cache`. Handles change less often than
thread state, and the walk is more expensive (a few MB buffer + a lot of
short-lived DuplicateHandle traffic). Cache is dropped when the pid
dies.

### Per-process network throughput (etw.rs)

Windows has no non-ETW API for "bytes sent/received by PID X." `GetIfTable2`
is per-adapter, not per-process; `GetPerTcpConnectionEStats` is TCP-only and
requires enabling ESTATS with admin. Task Manager doesn't show this column
because Task Manager doesn't start an ETW session; Resource Monitor does,
and auto-elevates.

We subscribe to `Microsoft-Windows-Kernel-Network` (GUID
`{7DD42A49-5329-4832-8DFD-43D979153A88}`), a manifest-based provider
registered by `tcpip.sys`. It fires a discrete event per TCP or UDP send
and receive on both IPv4 and IPv6. Event IDs used:

- 10 / 11 — TCP send / recv, IPv4
- 26 / 27 — TCP send / recv, IPv6
- 42 / 43 — UDP send / recv, IPv4
- 58 / 59 — UDP send / recv, IPv6

Each event carries a `size` (bytes) and a `PID` payload field. The `PID`
field is preferred over `EventRecord::process_id()` because receive events
often fire in DPC context where the header PID is the interrupted thread's
process (or zero), not the socket owner.

The `ferrisetw` crate wraps the whole StartTraceW / EnableTraceEx2 /
ProcessTrace dance behind a builder. The callback runs on a thread the
crate spawns for us; we accumulate into `Arc<Mutex<HashMap<u32, NetBytes>>>`
and the Tauri command thread takes cheap snapshots at 1 Hz to derive
bytes/sec deltas (same pattern as `sample_io`).

**Requires elevation.** Real-time user-mode ETW sessions can be created by
Administrators or members of the "Performance Log Users" group. Standard
users get `E_ACCESSDENIED` (0x80070005) from `StartTraceW`. When
`NetMonitor::start()` returns an error the app logs to stderr and runs
without the Net column populated — it never crashes.

**Stale sessions survive process crashes.** ETW session names live in the
kernel until stopped or reboot. If winglass is killed hard, a new instance
gets `ERROR_ALREADY_EXISTS` on start. We prepend our PID to the session
name and best-effort call `stop_trace_by_name` before starting, so
concurrent instances don't collide and orphaned sessions get cleaned up.

**Known caveats.**
- PID reuse within a poll interval mixes counters from the old and new
  process. Rare enough to ignore; would only misreport the very first
  delta after reuse.
- Cumulative `net_rx_total` / `net_tx_total` are since winglass start, not
  since process start. There is no easy way to backfill.

## Non-obvious API gotchas

### sysinfo 0.32
- `System::refresh_processes(All, true)` only samples CPU + memory. `exe`,
  `cmd`, `user_id`, `environ` come up empty. Must use
  `refresh_processes_specifics(All, true, ProcessRefreshKind::new().with_cpu().with_memory().with_exe(UpdateKind::OnlyIfNotSet).with_cmd(OnlyIfNotSet).with_user(OnlyIfNotSet).with_environ(OnlyIfNotSet))`.
- `ProcessRefreshKind::nothing()` does not exist. Use `::new()` (empty,
  opt-in) or `::everything()`.
- `sysinfo::MINIMUM_CPU_UPDATE_INTERVAL` is ~200 ms — too short for
  meaningful CPU% ranking. We use a 1 s tick.

### windows-rs 0.57
- `HWND` is `HWND(pub isize)`, not a raw pointer wrapper — use `HWND(0)`,
  never `HWND(null_mut())`.
- `HMODULE` implements `Param<HMODULE>` directly — pass `modules[i]`, not
  `Some(modules[i])`, to `GetModuleFileNameExW`.
- `HCATADMIN` / `HCATINFO` in the Catalog namespace are plain `isize`
  aliases, not exported as named types. Declare `let mut h_admin: isize = 0;`.
- `WINTRUST_DATA` / `WINTRUST_DATA_0` require **both**
  `Win32_Security_Cryptography` and `Win32_Security_WinTrust` features.
- `WTHelperProvDataFromStateData` and `WTHelperGetProvSignerFromChain`
  additionally need `Win32_Security_Cryptography_Sip`.
- `CryptCATAdminAcquireContext2` — pass `w!("SHA256")` explicitly for the
  hash algorithm. The default on modern Windows *is* SHA-256 but relying on
  it is fragile.
- NT APIs (`NtQuerySystemInformation`, `NtQueryInformationProcess`) live in
  `windows::Wdk::System::{SystemInformation, Threading}` and need
  `Wdk_System_SystemInformation` / `Wdk_System_Threading` features. Info
  classes are passed as `SYSTEM_INFORMATION_CLASS(88)` /
  `PROCESSINFOCLASS(61)` — the specific constants often aren't named.
- `QueryDosDeviceW` is in `Win32_Storage_FileSystem`, not
  `Win32_System_WindowsProgramming` (where MSDN docs suggest).

### ferrisetw 1.2
- `by_guid` accepts a hyphenated string like `"7DD42A49-...-3A88"`. Case
  is not sensitive.
- `stop_trace_by_name` is a **free function** in `ferrisetw::trace`, not
  an associated function on `UserTrace`. Easy to miss and the error
  message doesn't help.
- `TraceBuilder::start_and_process` spawns its own processing thread and
  returns immediately. Store the returned `UserTrace` to keep the trace
  alive — its `Drop` calls `ControlTrace(STOP)` for you.
- The callback bound is `FnMut + Send + Sync + 'static`. Capturing
  `Arc<Mutex<...>>` and calling `.lock()` inside is fine because the
  closure body is effectively `Fn` (it never mutates its captured
  environment, only the mutex's interior).

### Frontend
- Svelte 5 runes: DOM bindings via `bind:this` must go to `$state()`
  variables, not plain `let`, otherwise reactivity doesn't fire. Same for
  any variable read reactively.
- `getnameinfo` sockaddr lifetime bit us: when constructing the
  `SOCKADDR_IN` inside a `match` arm and returning a pointer to it, the arm
  scope ends and the pointer dangles before the API call. Keep both
  `SOCKADDR_IN` and `SOCKADDR_IN6` at outer scope, only branch on which one
  to fill.

### GeoIP database

Country badges on public remotes come from a db-ip.com IP-to-Country
Lite MMDB (~8 MB, CC BY 4.0). We treat it as a bundled resource:
`tauri.conf.json → bundle.resources = ["resources/*"]` puts the file next
to `winglass.exe` in the MSI/NSIS installer, and at startup we call
`app.path().resource_dir()` and open the DB with the `maxminddb` crate.
If the file is missing every lookup returns `None` and the badge simply
doesn't render.

**The file is not committed** (`src-tauri/resources/*.mmdb` is
`.gitignore`d). Download at build/dev time:

```powershell
$ym = Get-Date -Format "yyyy-MM"
$out = "src-tauri/resources/GeoLite2-Country.mmdb"
Invoke-WebRequest "https://download.db-ip.com/free/dbip-country-lite-$ym.mmdb.gz" -OutFile "$out.gz"
& "C:\Program Files\7-Zip\7z.exe" e "$out.gz" "-osrc-tauri/resources" -y
```

**Dev-only quirk:** Tauri's `resource_dir()` in `cargo run` mode
returns `target/debug/`, not `src-tauri/resources/`. Copy the file
there too, or the country badge stays empty in dev builds. The MSI
build handles the copy automatically.

## Roadmap

Nothing outstanding for v1.0.0 — the initial roadmap items are all in
the "Done" list below. Post-1.0 candidates (not committed, open for
discussion):

- Sortable/filterable columns in the Handles table (currently just
  histogram + row list).
- Persist the tree view collapse state across sessions (currently
  session-only in a SvelteSet).
- Signature revocation status — WinTrust returns it but we throw it
  away; would let us flag revoked signers as visibly as unsigned.
- Per-thread CPU% (we surface user + kernel time totals, not deltas).

**Done since initial MVP:**
- Per-process network throughput via ETW (Microsoft-Windows-Kernel-Network,
  `ferrisetw` crate). Requires admin. See "Per-process network throughput"
  above.
- Top-bar chip + detail-panel hint when `NetMonitor` failed to start, so
  the empty Net column is self-explanatory. Backed by a
  `net_monitor_active` Tauri command.
- `k` opens the kill confirmation prompt in the detail panel; `Enter`
  confirms.
- Per-process threads with wait reasons — collapsible section in the
  detail panel showing TID, state, wait reason, CPU time, context
  switches. See "Threads" above.
- Per-process open handles with resolved object names via a bounded
  sacrificial worker (50 ms per-call timeout, 2 s global budget). See
  "Handles" above.
- Filter chips in the top bar (unsigned / Windows-OS-signed / network
  activity), keyboard-toggleable with `u` / `w` / `n`. `sigKind()`
  centralises the classification the badge color, badge title, stats
  counter, main-row text color, and filter predicates all switch on.
- Per-row CPU sparkline in the process table (replaces the old
  horizontal progress bar), backed by the same 60-sample VecDeque as
  the detail-panel graph.
- Process tree view (`t` keyboard toggle) — build a hierarchy from
  `parent_pid`, chevrons per node, sibling-level sort. Filters
  preserve ancestors of matched rows, ancestor-only rows dimmed to
  50 %.
- GeoIP country badges on public remote IPs in the connection list.
  See "GeoIP database" above.

## Anti-goals

Not everything belongs in winglass. These are out of scope by design:

- **Auto-updater.** Adds attack surface, no maintenance appetite for it.
- **Cloud sync / telemetry.** Violates the zero-external-services principle.
- **Kernel driver to inspect PPL DLLs.** WHQL signing costs $, ongoing
  compliance overhead, and turns this from a hobby tool into a compliance
  project.
- **Anti-malware behavior.** winglass surfaces what's on the system; it does
  not act on threats.
- **Task management for regular users.** This is a tool for developers,
  security analysts, and admins. If a user's first question is "why do I
  need this?", it's not for them.

## Design principles

- **Signature is truth.** The UI never trusts a process by its path or name.
  A row's color comes from its exe's signer CN, nothing else.
- **Zero external services.** No cloud calls, no telemetry, no auto-update
  reaching out. Reverse DNS uses the OS resolver.
- **Read-only by default.** The only mutating action is Kill Process, and
  it requires explicit confirmation.
- **Owned state on the Rust side.** The frontend renders what it's told;
  it doesn't maintain a parallel model.
- **Cheap enough to poll.** All commands complete in under 200 ms; refresh
  is 1 Hz. If a feature can't fit that budget, it goes in a worker.

## Working style

- Small, atomic changes. If a change requires editing more than three files
  in one commit, split it.
- Test in the running dev server (`npm run tauri dev`) before considering
  anything done. Type checks pass ≠ feature works.
- Rust changes trigger an incremental cargo rebuild (~30 s); Svelte changes
  hot-reload. Batch Rust edits before rebuild.
- Update this file when architecture shifts. Don't let it decay.
